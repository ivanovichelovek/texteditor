# Roadmap

> Этот план собран из **текущей структуры репозитория** и трёх зафиксированных решений
> (модальный vim-like редактор; весь файл в одном `gap_buffer` с символами `'\n'`;
> начинаем со слоя `editor/`). В прошлых разговорах отдельного обсуждения плана не было —
> так что это **выведенный** план, а не восстановленный. Пункты, помеченные ⚠️ — это
> принятые допущения, которые можно пересмотреть.

## Где мы сейчас

| Слой            | Файлы                   | Статус                                                                                                      |
| --------------- | ----------------------- | ----------------------------------------------------------------------------------------------------------- |
| `buffer/`       | `gap_buffer`            | ✅ Готов: SSO, кастомные аллокаторы, полный набор итераторов, exception-safety, ~100 тестов, coverage + ASan |
| `editor/`       | `editor.{hpp,cpp}`      | ⬜ Пустые заглушки — **ближайший фокус**                                                                     |
| `terminal/`     | `input.*`, `renderer.*` | ⬜ Пустые заглушки                                                                                           |
| `main.cpp`      | —                       | ⬜ Пустой (бинарник не слинкуется, пока нет `main()`)                                                        |
| `benchmarks/`   | `buffer_benchmark.cpp`  | ⬜ Пустой                                                                                                    |
| `tests/editor/` | —                       | ⬜ Директории ещё нет                                                                                        |

Правило зависимостей: `terminal → editor → buffer`. Нижний слой ничего не знает о верхних.

## Ключевые проектные решения (зафиксированы)

1. **Курсор = позиция gap'а.** API буфера уже это диктует: `step()`,
   `erase_in_front_of_cursor()`, `erase_in_back_of_cursor()`, `get_cursor_position_index()`
   — все работают от gap-индекса. Отдельного «курсора» проектировать не нужно: курсор
   редактора *и есть* gap. Слой `editor/` лишь наводит поверх этого логическую адресацию.
2. **Весь файл — один плоский `gap_buffer<char>`**, переносы строк `'\n'` — обычные символы.
   Логическая позиция `(row, col)` вычисляется поверх буфера.
3. ⚠️ **Осознанный компромисс модели.** Любой запрос `(row, col)`, поиск границ строки для
   отрисовки и вертикальные движения `j`/`k` — это **O(n) скан / сдвиг gap'а**. Для MVP
   приемлемо. Оптимизация (кэш смещений строк / индекс строк) вынесена в фазу 4, а не
   считается бесплатной.

## Фаза 1 — Слой `editor/` (ближайший фокус)

Слой чистой логики: тестируется под gtest без UI. Именно тесты — основной цикл обратной
связи, пока нет ничего запускаемого.

### M1 — Каркас и адресация строк
- [ ] `class editor`, владеющий `gap_buffer`.
- [ ] `enum class Mode { Normal, Insert, Visual }` + текущий режим.
- [ ] Отображение gap-индекса ↔ логическая `(row, col)`: `cursor_row()`, `cursor_col()`.
- [ ] Строковые утилиты поверх буфера: `line_count()`, `line_start(row)`, `line_end(row)`,
      `current_line()`.
- [ ] Тесты: маппинг позиции для текста с `'\n'`, пустой буфер, курсор в начале/конце строки.

### M2 — Движения (normal mode)
- [ ] Горизонталь: `h` `l` `0` `$`, стрелки ←→.
- [ ] Вертикаль: `j` `k`, стрелки ↑↓ — с «липкой колонкой» (desired column).
- [ ] По словам: `w` `b` `e`.
- [ ] По файлу: `gg` `G`.
- [ ] Реализация через `buffer.step()` + навигацию с учётом `'\n'`.
- [ ] Тест на каждое движение, включая граничные случаи (короткая строка при вертикали).

### M3 — Insert mode и правки
- [ ] Вход в insert: `i` `a` `o` `O` (с корректной позицией).
- [ ] В insert: печатные символы → `insert`, Backspace → `erase_in_front_of_cursor`,
      Delete → `erase_in_back_of_cursor`, Enter → `insert('\n')`, Esc → normal.
- [ ] Правки из normal: `x` (символ), `dd` (строка).
- [ ] Тесты.

### M4 — Visual mode и операторы
- [ ] `v` — начало выделения, движения расширяют его.
- [ ] Операторы над выделением: `d` `y` `c`.
- [ ] Регистр (простой `std::string`) + вставка `p` `P`.
- [ ] Тесты.

### M5 — Command-line
- [ ] `:w` `:q` `:wq` — мост к файловому I/O (см. фазу 3).
- [ ] Разбор command-line строки, сообщения об ошибках.

## Фаза 2 — Слой `terminal/`

- [ ] **T1 — raw-режим:** `termios` (отключить echo/canonical), alt-screen, гарантированное
      восстановление терминала при выходе/исключении (RAII-обёртка).
- [ ] **T2 — декодирование ввода:** байты → key-события, включая escape-последовательности
      (стрелки, Esc, спецклавиши). Питает `editor`.
- [ ] **T3 — рендерер:** отрисовка видимой области буфера, позиционирование курсора,
      статус-строка с текущим режимом. (Diff-отрисовка — позже.)
- [ ] **T4 — главный цикл** в `main.cpp`: read → `editor.handle(key)` → render.

## Фаза 3 — Файловый I/O

- [ ] Загрузка файла в `gap_buffer` (конструктор из `std::string` уже есть).
- [ ] Сохранение на диск (`:w`), обработка ошибок/прав доступа.
- [ ] Аргумент командной строки: `texteditor <file>`.

## Фаза 4 — Производительность и полировка

- [ ] Заполнить `benchmarks/buffer/buffer_benchmark.cpp` (Google Benchmark): insert, step,
      reserve на разных размерах.
- [ ] ⚠️ Кэш смещений строк / индекс строк — снять O(n) с навигации по `(row, col)`.
- [ ] Diff-отрисовка терминала (перерисовывать только изменившиеся ячейки).
- [ ] Undo/redo (журнал операций или снапшоты).

## Технический долг в `buffer/` (по ходу дела)

      — знаковое/беззнаковое сравнение (`buffer.cpp:441`). Аналогично `step()` (`:482`).
- [ ] Держать `tests/gap_buffer_iterator_regression` в актуальном состоянии — там уже
      копятся кейсы прошлых багов итератора (обход gap'а, subscript на границе gap'а).
