# Roadmap

> Этот план собран из **текущей структуры репозитория** и трёх зафиксированных решений
> (модальный vim-like редактор; весь файл в одном `gap_buffer` с символами `'\n'`;
> начинаем со слоя `editor/`). В прошлых разговорах отдельного обсуждения плана не было —
> так что это **выведенный** план, а не восстановленный. Пункты, помеченные ⚠️ — это
> принятые допущения, которые можно пересмотреть.

## Где мы сейчас

| Слой            | Файлы                   | Статус                                                                                                      |
| --------------- | ----------------------- | ----------------------------------------------------------------------------------------------------------- |
| `buffer/`       | `gap_buffer`            | ✅ Готов: SSO, кастомные аллокаторы, полный набор итераторов, exception-safety, ~100 тестов, coverage + ASan |
| `editor/`       | `editor.{hpp,cpp}`      | ⬜ Пустые заглушки — **ближайший фокус**                                                                     |
| `terminal/`     | `input.*`, `renderer.*` | ⬜ Пустые заглушки                                                                                           |
| `main.cpp`      | —                       | ⬜ Пустой (бинарник не слинкуется, пока нет `main()`)                                                        |
| `benchmarks/`   | `buffer_benchmark.cpp`  | ⬜ Пустой                                                                                                    |
| `tests/editor/` | —                       | ⬜ Директории ещё нет                                                                                        |

Правило зависимостей: `terminal → editor → buffer`. Нижний слой ничего не знает о верхних.

## Ключевые проектные решения (зафиксированы)

1. **Построчная (двухуровневая) модель — гибрид.** Весь файл хранится как
   `gap_buffer<std::string>` (внешний gap по строкам), а строка, которую сейчас правим,
   «раскрывается» в отдельный активный `gap_buffer<char>`. Это даёт O(1) на навигацию,
   O(1) аморт. на вставку/удаление строк у курсора и O(len) только на активацию
   редактируемой строки. Плоский `gap_buffer<char>` со всем файлом и `'\n'` **отвергнут**:
   для цели редактора кода (LSP оперирует `{line, character}`, подсветка/отрисовка —
   построчные) он требует O(n)-сканов, которые пришлось бы гасить синхронизируемым сбоку
   индексом строк. `gap_buffer<gap_buffer<char>>` (чистый «двумерный») даёт то же, что
   гибрид, но дороже: два уровня gap-инвариантов и вложенная SSO. См. [[buffer-model-decision]].
2. **Курсор = позиция gap'а.** API буфера это диктует (`step()`,
   `erase_in_front_of_cursor()`, `get_cursor_position_index()` — все от gap-индекса).
   В гибриде: строка курсора = позиция внешнего gap; колонка = позиция gap внутри
   активного `gap_buffer<char>`.
3. **Темплейтизация `gap_buffer` по `T` (объём работ).** Сейчас класс захардкожен под
   `char`. Для внешнего `gap_buffer<std::string>` нужен один темплейт-проход:
   char-специфичные конструкторы (`const char*`, `strlen`, `to_string`) ограничить через
   `requires`/специализацию для `T=char`; `kSSOBufferSize` пересчитать под «тяжёлый»
   элемент. `T=std::string` заметно проще, чем `T=gap_buffer<char>` (нормальный value-тип,
   без вложенной SSO и второго уровня gap-инвариантов).

## Фаза 1 — Слой `editor/` (ближайший фокус)

Слой чистой логики: тестируется под gtest без UI. Именно тесты — основной цикл обратной
связи, пока нет ничего запускаемого.

### M0 — Темплейтизация `gap_buffer` по `T`
- [ ] Обобщить `gap_buffer<char>` → `gap_buffer<T>` (union/SSO, конструкторы, итераторы).
- [ ] Char-специфичные конструкторы (`const char*`, `strlen`, `to_string`) — под `requires`
      / специализацию для `T=char`, чтобы `gap_buffer<std::string>` их не тянул.
- [ ] Пересчитать `kSSOBufferSize` под «тяжёлый» элемент (или сделать зависимым от `T`).
- [ ] Прогнать существующие тесты `gap_buffer<char>` без регрессий + добавить кейсы для
      `gap_buffer<std::string>` (в т.ч. move-only/exception-safety на нетривиальном `T`).

### M1 — Каркас и построчная адресация
- [ ] `class editor`: внешний `gap_buffer<std::string>` (строки) + активный
      `gap_buffer<char>` для строки под курсором (см. lifecycle ниже).
- [ ] `enum class Mode { Normal, Insert, Visual }` + текущий режим.
- [ ] Курсор `(row, col)`: `row` = позиция внешнего gap, `col` = позиция gap активной
      строки. Обе — O(1), без сканов по `'\n'`.
- [ ] Lifecycle активной строки: активировать `lines[r]` в char-буфер при первой правке
      (O(len)); коммитить обратно `to_string()` при уходе со строки / выходе из insert.
- [ ] Утилиты: `line_count()`, `line(row)`, `current_line()`.
- [ ] Тесты: адресация `(row, col)`, пустой буфер, курсор в начале/конце строки,
      активация/коммит строки не теряет и не дублирует содержимое.

### M2 — Движения (normal mode)
- [ ] Горизонталь: `h` `l` `0` `$`, стрелки ←→.
- [ ] Вертикаль: `j` `k`, стрелки ↑↓ — с «липкой колонкой» (desired column).
- [ ] По словам: `w` `b` `e`.
- [ ] По файлу: `gg` `G`.
- [ ] Реализация через `buffer.step()` + навигацию с учётом `'\n'`.
- [ ] Тест на каждое движение, включая граничные случаи (короткая строка при вертикали).

### M3 — Insert mode и правки
- [ ] Вход в insert: `i` `a` `o` `O` (с корректной позицией).
- [ ] В insert: печатные символы → `insert`, Backspace → `erase_in_front_of_cursor`,
      Delete → `erase_in_back_of_cursor`, Enter → `insert('\n')`, Esc → normal.
- [ ] Правки из normal: `x` (символ), `dd` (строка).
- [ ] Тесты.

### M4 — Visual mode и операторы
- [ ] `v` — начало выделения, движения расширяют его.
- [ ] Операторы над выделением: `d` `y` `c`.
- [ ] Регистр (простой `std::string`) + вставка `p` `P`.
- [ ] Тесты.

### M5 — Command-line
- [ ] `:w` `:q` `:wq` — мост к файловому I/O (см. фазу 3).
- [ ] Разбор command-line строки, сообщения об ошибках.

## Фаза 2 — Слой `terminal/`

- [ ] **T1 — raw-режим:** `termios` (отключить echo/canonical), alt-screen, гарантированное
      восстановление терминала при выходе/исключении (RAII-обёртка).
- [ ] **T2 — декодирование ввода:** байты → key-события, включая escape-последовательности
      (стрелки, Esc, спецклавиши). Питает `editor`.
- [ ] **T3 — рендерер:** отрисовка видимой области буфера, позиционирование курсора,
      статус-строка с текущим режимом. (Diff-отрисовка — позже.)
- [ ] **T4 — главный цикл** в `main.cpp`: read → `editor.handle(key)` → render.

## Фаза 3 — Файловый I/O

- [ ] Загрузка файла: разбить на строки → `gap_buffer<std::string>` (по строке на элемент).
- [ ] Сохранение на диск (`:w`) — склейка строк через `'\n'`; обработка ошибок/прав доступа.
- [ ] Аргумент командной строки: `texteditor <file>`.

## Фаза 4 — Производительность и полировка

- [ ] Заполнить `benchmarks/buffer/buffer_benchmark.cpp` (Google Benchmark): insert, step,
      reserve на разных размерах; отдельно — вставка/удаление строк во внешнем буфере.
- [ ] Diff-отрисовка терминала (перерисовывать только изменившиеся ячейки).
- [ ] Undo/redo (журнал операций или снапшоты).
- [ ] ⚠️ Профиль памяти на файлах с миллионами коротких строк (overhead объекта-строки).
      Если станет узким местом — смотреть в сторону rope / piece-tree (см. [[buffer-model-decision]]).

## Фаза 5 — Редактор кода (дальний горизонт)

Цель проекта — со временем превратить `texteditor` из простого текстового редактора в
**редактор кода**. Построчная модель хранения выбрана в т.ч. под эту цель: `(row, col)`
адресация нативна, а подсветка/отрисовка/LSP — построчные.

- [ ] **Подсветка синтаксиса** — токенизатор/лексер построчно; кэш состояния лексера на
      строку, перекрашивать только видимые строки (адрес строки уже O(1)).
- [ ] **Большие файлы** — навигация по `(row, col)` уже O(1); следить за памятью
      (см. пункт в фазе 4).
- [ ] Авто-отступ, парные скобки, поиск/замена (в т.ч. по проекту).
- [ ] ⚠️ **LSP-клиент** (максимум амбиций) — автодополнение, диагностика, переходы.

Проектная заметка: при разработке слоёв `editor/` и рендерера не закладывать допущение
«только plain text» — поверх буфера позже ляжет синтаксический слой.
